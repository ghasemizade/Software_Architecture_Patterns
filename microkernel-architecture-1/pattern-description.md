# تحلیل الگو



```
قراردادهای بین ماژول های پلاگین و سیستم اصلی می توانند
از قراردادهای استاندارد گرفته تا قراردادهای سفارشی را شامل می شود. سفارشی
قراردادها معمولاً در موقعیت‌هایی یافت می‌شوند که اجزای پلاگین
ننت ها توسط شخص ثالثی توسعه می یابند که شما هیچ کنترلی روی آن ندارید
قرارداد استفاده شده توسط افزونه. در چنین مواردی، ایجاد
یک آداپتور بین تماس پلاگین و اتصال استاندارد خود خوردید
به طوری که سیستم اصلی برای هر کدام نیازی به کد تخصصی نداشته باشد
پلاگین. هنگام ایجاد قراردادهای استاندارد (معمولاً اجرا می شود
از طریق XML یا نقشه جاوا)، مهم است که ایجاد را به خاطر بسپارید
یک استراتژی نسخه سازی از همان ابتدا.
```

```
ماژول های پلاگین را می توان از طریق یک به سیستم هسته متصل کرد
راه های مختلف، از جمله OSGi (ابتکار دروازه سرویس باز)،
پیام رسانی، خدمات وب یا حتی اتصال مستقیم نقطه به نقطه (به عنوان مثال،
نمونه سازی شی). نوع اتصالی که استفاده می کنید به آن بستگی دارد
نوع برنامه ای که می سازید (محصول کوچک یا شرکت بزرگ
برنامه کاربردی) و نیازهای خاص شما (مانند استقرار یا عدم توزیع
معماری میکروکرنل
استقرار خراجی). خود الگوی معماری اینطور نیست
هر یک از این جزئیات پیاده سازی را مشخص کنید، فقط افزونه
ماژول ها باید مستقل از یکدیگر باقی بمانند.
```

```
سیستم اصلی باید بداند که کدام ماژول های پلاگین هستند
موجود و نحوه دسترسی به آنها یکی از راه های رایج پیاده سازی
انجام این کار از طریق نوعی رجیستری پلاگین است. این رجیستری
اطلاعات مربوط به هر ماژول پلاگین، از جمله مواردی مانند
نام، قرارداد داده، و جزئیات پروتکل دسترسی از راه دور (بستگی به
نحوه اتصال افزونه به سیستم اصلی). برای امتحان-
ple، افزونه ای برای نرم افزار مالیاتی که موارد ممیزی مالیاتی پرخطر را علامت گذاری می کند
ممکن است یک ورودی رجیستری داشته باشد که حاوی نام است
سرویس (AuditChecker)، قرارداد داده (داده های ورودی و خروجی).
داده ها) و قالب قرارداد (XML). همچنین ممکن است حاوی یک WSDL باشد
(زبان تعریف خدمات وب) در صورت دسترسی به افزونه
از طریق SOAP
```

```
ماژول های پلاگین، اجزای مستقل و مستقلی هستند که
شامل پردازش تخصصی، ویژگی های اضافی و کد سفارشی است
که به منظور تقویت یا گسترش سیستم اصلی برای تولید مواد افزودنی است.
قابلیت های تجاری ملی به طور کلی، ماژول های پلاگین باید باشند
مستقل از سایر ماژول های پلاگین، اما مطمئناً می توانید طراحی کنید
پلاگین هایی که نیاز به وجود افزونه های دیگر دارند. در هر صورت، اینطور است
مهم است که ارتباط بین پلاگین ها را در حد کوچک نگه دارید
مادر برای جلوگیری از مسائل وابستگی
```

```
سیستم اصلی سنت الگوی معماری میکروکرنل -
ally تنها دارای حداقل عملکرد مورد نیاز برای ساخت است
سیستم عملیاتی بسیاری از سیستم عامل ها میکرو
الگوی معماری هسته، از این رو منشاء نام این الگو است.
از منظر برنامه تجاری، سیستم اصلی اغلب است
به عنوان منطق تجاری عمومی بدون کد سفارشی برای ویژه تعریف شده است

```
